① 토대Workflow vs Agent, 그리고 5개 합성 패턴

Anthropic은 먼저 둘을 가른다 — Workflow는 "LLM과 도구가 사전 정의된 코드 경로로 오케스트레이션되는 시스템", Agent는 "LLM이 스스로 프로세스와 도구 사용을 동적으로 제어하는 시스템".1

🪶 가장 중요한 권고: 단순성

Anthropic은 복잡한 프레임워크보다 단순성을 명시적으로 권한다. "LLM API를 직접 써서 시작하라 — 많은 패턴은 몇 줄의 코드로 구현된다", "가장 단순한 해법을 찾고, 명백히 결과가 좋아질 때만 복잡도를 더하라". 프레임워크는 프롬프트를 가리고 디버깅을 어렵게 하는 추상화를 더한다.1

③ 근본 제약 다루기컨텍스트 엔지니어링 4대 기법

"컨텍스트는 근본적 제약"이라는 전제 위에서, Anthropic은 토큰을 아끼는 구체 기법을 제시한다.4

① Just-in-Time 검색

모든 데이터를 사전 처리하지 않고, 에이전트가 경량 식별자(파일경로·쿼리·링크)만 들고 있다가 런타임에 도구로 동적 로드. RAG-first의 의도적 역전 — Anthropic은 grep/tail 같은 agentic search를 기본값으로, semantic search는 속도가 필요할 때만 추가하라고 권한다.43

② Sub-Agent 격리

전문화된 서브에이전트가 깨끗한 컨텍스트 윈도우로 깊은 작업을 하고, lead에는 1,000–2,000 토큰의 압축 요약만 반환. 상세 탐색 맥락은 서브에이전트 안에 격리되어 메인이 종합에 집중.4

③ 외부 메모리 / 진행 로그

완료 단계를 요약해 외부 메모리에 저장(프로덕션 배포 기법). 장기 실행 에이전트는 claude-progress.txt + git 히스토리로 fresh 컨텍스트에서도 작업 상태를 빠르게 복원한다.65

④ Progressive Disclosure (Skills)

스킬 본문은 "쓸 때만" 로드 — 긴 참고자료도 필요 전까지 토큰 비용 거의 0. context: fork를 주면 스킬이 격리 서브에이전트로 실행되어 대화 히스토리에 접근하지 않는다(스킬↔서브에이전트 연결).7

④ 관점 1 · 제품 개발에이전틱 코딩: 권장 4단계 + 장기실행 하네스

Claude Code의 공식 권장 워크플로우는 연구/계획을 실행에서 분리해 "잘못된 문제를 푸는 것"을 막는 4단계 루프다.2

장기 실행(시간·일 단위) 작업의 하네스 패턴

핵심 난제는 "에이전트가 매 세션을 이전 기억 없이 시작한다"는 것(교대 근무 비유). Anthropic의 해법은 구체적인 2-에이전트 구조다.513

🛠️Initializer Agent

첫 1회만 실행 — 환경 셋업: init.sh, 진행 로그 claude-progress.txt, 최초 git 커밋.

⌨️Coding Agent

이후 매 세션 — 점진적 진행 후 다음 세션을 위한 구조화된 artifact(로그·커밋)를 남긴다.

⚠ 막아야 할 실패 모드: "테스트 없이 완료 표시"

Claude가 적절한 테스트 없이 기능을 '완료'로 마킹하는 경향이 있다. Anthropic의 처방: ① 테스팅 도구 제공(Puppeteer MCP) — 성능을 극적으로 개선 ② 요구사항을 feature_list.json으로 두고 모델은 passes 필드 상태만 변경 허용("테스트 삭제·편집 금지", "신중히 테스트한 뒤에만 passing 표시"). 이것이 검증 루프를 구조로 강제하는 방식.5

⑤ 관점 2 · 보편 업무리서치·문서·분석에서의 같은 골격

비개발 업무도 동일한 orchestrator-worker + 검증 구조를 쓴다. Anthropic의 멀티에이전트 리서치 시스템이 대표 예시이며, 이 문서 자체가 그 패턴(분해→병렬 검색→압축 요약→종합→적대적 검증)으로 생성되었다.6

업무orchestrator-worker 적용검증 루프(검증자)
리서치lead가 질문 분해 → 3–5개 서브에이전트 병렬 검색 → 각 1–2k 토큰 요약 반환 → lead 종합주장별 다수 출처 교차검증 · 적대적 검증(반증 시 기각)
문서 작성개요(=CLAUDE.md식 베이스라인) → 섹션 병렬 초안 → 통합fresh-context 리뷰어가 원문 대조
분석가설 분해 → just-in-time으로 데이터 동적 로드 → 계산 위임수치 재현·역산, 외부 메모리에 중간결과 저장

🧩 스킬은 비개발에도 그대로

Agent Skills는 코딩 전용이 아니다 — Anthropic은 "실세계 업무를 위한 에이전트 장비"로 스킬을 제시하고, Claude Cowork 등 일반 업무 제품에 동일 메커니즘(SKILL.md·progressive disclosure)이 적용된다.101112 (단, 비개발 시나리오의 독립 효과 검증은 아직 코딩 대비 빈약 — ⑦ 참조)

⑥ 개념 관계도맥락 제약을 중심으로 도는 전체 시스템

중앙의 "컨텍스트 제약"에서 모든 기능이 파생된다. 바깥 고리는 실행 루프(gather→act→verify→repeat).

⑦ 현실 점검vendor 편향과 기각된 주장 (정직성)

✕ 검증 과정에서 기각된 2개 주장

1) "멀티에이전트가 단일 에이전트 대비 90.2% 우수" — 1:2로 기각. Anthropic 내부 eval 단일 출처라 독립 검증 불가. 이 수치는 본 문서 어디에도 쓰지 않았다.6

2) "harness의 일반적 정의(여러 컨텍스트 윈도우에 걸친 오케스트레이션 프레임워크)" — 3:0 기각. 출처 근거가 일반화를 지지하지 않아, 본 문서는 harness를 Anthropic 1차 자료가 실제로 서술한 범위(장기실행·다중세션·Agent SDK 기반)로만 한정해 기술했다.5

⚠ Vendor 편향 고지

이 주제는 본질적으로 Anthropic이 권위 출처다(방법론·도구 정의는 만든 쪽이 1차 자료). 그래서 발견의 대부분이 서술적·정의적·권고적 주장이며 성능 우위 주장이 아니어서 편향이 정확성을 크게 훼손하진 않는다. 그럼에도 모든 핵심 항목을 독립 3자 출처(Spring AI, ZenML, ByteByteGo, DataCamp, Lee Hanchung, Victor Dibia 등)로 교차검증했다.

🗣️ 균형을 위한 독립·비판적 시각

2026년 들어 "Claude Code가 느려지고 품질이 떨어졌다"는 실무자 보고, 비용 관련 마이크로소프트의 행보 등 비판적 목소리도 존재한다. 방법론(개념·구조)은 유효하되, 도구의 체감 품질·비용은 시점에 따라 변동하므로 별도로 추적할 것.192021

🔎 남은 열린 질문

  • 최신 모델(Opus 4.8 등) 기준 멀티에이전트 정량 이득의 검증된 재측정치가 있는가 (기각된 90.2% 대체).
  • plan mode·output styles·worktree·headless 등은 1차 출처 매핑이 더 필요 — 본 문서는 verified 집합만 단정.
  • 스킬의 context:fork 격리가 비개발 업무에서도 코딩만큼 효과적이라는 독립 검증.
  • "단순성 우선" 권고와 점점 복잡해지는 SDK/harness 사이의 임계 기준.
