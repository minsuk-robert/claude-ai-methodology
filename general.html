AI를 가장 잘 쓰는 방법론

제품 개발과 보편 업무, 두 관점에서 본 실행 프레임워크 — 그리고 그 속에 등장하는 서브에이전트·하네스·컨텍스트 엔지니어링·오케스트레이션·검증 루프가 실제로 어떻게 맞물리는지에 대한 구체적 해설.

핵심 명제. 잘 쓴다는 것은 더 똑똑한 모델을 고르는 문제가 아니라 컨텍스트(맥락) 엔지니어링 → 작업 분해·오케스트레이션 → 검증 루프 → 인간 감독이라는 시스템을 설계하는 문제다.1,2 단, AI는 만능 가속기가 아니라 증폭기(amplifier)에 가깝다 — 좋은 시스템 위에선 크게, 나쁜 시스템 위에선 미미하게(때로 마이너스로) 작동한다.6,8

🧩Context Engineering

매 추론 단계에서 한정된 컨텍스트 창에 "어떤 토큰을 채울지" 큐레이션하는 기술. 프롬프트 작성을 넘어, 도구·기록·외부지식·이전 결과를 최적 구성으로 관리한다.
적용 · 컨텍스트는 유한 자원 — 길어질수록 성능이 떨어지는 "context rot"가 존재. Anthropic은 Compaction(요약 압축)·Note-Taking(외부 메모)·Sub-Agent(맥락 격리) 3기법을 권장.1

⚙️Harness / 하네스

모델 자체가 아니라 모델을 감싸 동작하게 만드는 실행 골격. 도구 정의·권한·루프·메모리·중단/재시작·관측을 담당하는 "엔진룸".
적용 · 같은 모델이라도 하네스 설계가 결과를 가른다. Claude Code·Cursor 등은 본질적으로 잘 만든 하네스다. 하네스가 컨텍스트 엔지니어링과 검증 루프를 강제·자동화한다.2

🛰️Sub-Agent / 서브에이전트

메인 에이전트가 호출하는 독립된 컨텍스트·목표를 가진 하위 작업자. 자기 작업만 보고 "결론"만 돌려줘 메인 맥락을 깨끗하게 유지.
적용 · 탐색·검색처럼 토큰을 많이 먹는 일을 격리해 위임 → 메인은 의사결정에 집중. 컨텍스트 엔지니어링의 3대 기법 중 하나이자 멀티에이전트의 단위.1,3

🎼Orchestration

리드(오케스트레이터)가 작업을 분해 → 여러 워커에 병렬 분배 → 결과 종합하는 조정 구조. 동기·비동기, 직렬·병렬을 설계한다.
적용 · "orchestrator-workers" 패턴이 가장 보편적 출발점. 리드가 계획을, 서브에이전트가 실행을 담당.3,4

👥Multi-Agent

여러 에이전트가 역할을 나눠 협업하는 시스템. 분해·병렬·종합으로 단일 에이전트의 컨텍스트·관점 한계를 넘는다.
적용 · 폭넓은 탐색형 과제에 강함. 단, 토큰 비용이 급증하고(아래 차트) 실패 모드도 늘어 항상 우월하진 않다.3,7

🔁Eval / Verification Loop

생성한 결과를 별도 평가자가 채점·피드백하고 다시 고치는 순환. Evaluator-Optimizer = Generator-Verifier(동일 개념).
적용 · "성공 기준이 명확할 때" 가장 효과적. 코드는 테스트·리뷰가 검증자, 비개발은 외부 출처 대조가 검증자. 인간 리뷰는 여전히 필수.4,5

② 토대Anthropic의 5가지 에이전트 패턴 + Workflow vs Agent

방법론을 만들기 전, 먼저 "워크플로우(정해진 코드 경로)"와 "에이전트(모델이 스스로 경로를 정함)"를 구분하는 것이 출발점이다. 그 위에 5가지 조합 패턴이 있다.2

패턴구조언제 쓰나
1. Prompt Chaining
직렬 연결
출력→다음 입력으로 단계 분할작업을 고정된 하위 단계로 깔끔히 쪼갤 수 있을 때
2. Routing
분류·분기
입력을 유형 분류 후 전용 경로로유형별 처리가 명확히 다를 때(문의 분류 등)
3. Parallelization
병렬화
분할(sectioning)·다수결(voting)독립 하위작업 동시 처리 / 다관점 교차검증
4. Orchestrator–Workers
오케스트레이션 ★
리드가 동적으로 분해→워커 위임→종합하위작업을 미리 못 정하는 복잡·개방형 과제
5. Evaluator–Optimizer
검증 루프 ★
생성↔평가 순환으로 점진 개선명확한 평가 기준이 있고 반복이 품질을 올릴 때

★ 표시(4·5)가 본 방법론의 두 축. "단순한 것부터" 원칙 — 단일 프롬프트로 충분하면 멀티에이전트를 쓰지 말고, 필요할 때만 orchestrator-subagent 패턴으로 시작하라는 것이 Anthropic의 권고.2,4

④ 관점 2 · 보편 업무비개발 지식근로의 체계적 AI 활용

리서치·문서·기획·영업·분석에서도 동일한 골격(분해→실행→검증→감독)이 적용된다. 다만 "검증자"가 테스트가 아니라 외부 출처 대조와 명시적 성공 기준이라는 점이 다르다.5

업무분해 / 실행(서브에이전트적 사용)검증 루프(무엇이 검증자인가)
리서치질문을 여러 각도로 쪼개 병렬 검색 → 소스 수집 → 종합(본 보고서가 그 예시)주장별 다수 출처 교차검증 · 1차 출처 우선
문서 작성개요(스펙) → 섹션별 초안 → 통합 → 톤 정리사실 체크리스트 · 원문 인용 대조
기획 / 전략옵션 N개를 독립 생성 → 평가 패널이 채점 → 승자 종합명시적 평가 기준(MECE·So-What) · 반대 관점 에이전트
영업 / 고객세그먼트별 메시지 변형 병렬 생성 → 라우팅실제 응답률·도메인 사실 검증
데이터 분석가설 분해 → 쿼리/계산 위임 → 시각화수치 재현·역산 · 이상치 점검

비개발 영역의 핵심 리스크: 비판적 사고의 위축

마이크로소프트·CMU의 CHI 2025 연구(지식근로자 319명)에 따르면, AI에 대한 신뢰가 높을수록 사용자의 비판적 사고가 줄고, 반대로 자기 능력에 대한 확신이 높을수록 비판적 사고가 늘었다. 즉 AI는 루틴 인지 노력을 덜어주지만, "검증을 사람이 쥐지 않으면" 품질이 조용히 무너진다.5

실무 처방: ① 작업 전 성공 기준을 사람이 먼저 정의 ② 결과는 외부 출처로 대조 ③ AI를 "최종 답"이 아니라 "초안 + 검증 대상"으로 취급.

⑤ 실제 수행 방침아이디어를 결과물로 바꾸는 작업 운영 매뉴얼

실행 단계에서는 AI에게 "좋게 해줘"가 아니라 작업 계약·권한·검증 기준·종료 조건을 준다. 단일 프롬프트로 끝나는 일은 단일 루프로, 정보량·관점·독립성이 커질 때만 병렬/서브에이전트로 승격한다.2,25,26

1. 작업 계약서부터 고정

모든 실행은 10분짜리 작업 계약서로 시작한다. 계약서가 없으면 모델은 목표를 추측하고, 검증자는 무엇을 통과시켜야 하는지 모른다.

  • 목표/비목표: 이번 라운드에서 만들 것과 일부러 하지 않을 것
  • 입력 자료: 신뢰 가능한 원문, 코드 위치, 고객 메모, 데이터 범위
  • 산출물 형식: HTML 섹션, PR, 표, 보고서, 테스트 결과처럼 제출 단위를 명시
  • 성공 기준: 사실 정확도, 테스트 통과, 스크린샷 확인, 고객 승인 등 측정 가능한 게이트
  • 권한/금지: 삭제, 결제, 고객 발송, 배포, 개인정보 처리처럼 승인 없이는 못 하는 일

2. 위임 레벨을 먼저 정한다

레벨AI가 할 수 있는 일사람 게이트
L0 초안아이디어, 구조, 문장 후보사람이 전부 선택
L1 보조 실행정해진 파일 읽기, 표 정리, 단순 수정diff 확인
L2 격리 작업분리된 파일/브랜치/워크트리에서 구현테스트·리뷰 통과
L3 병렬 에이전트서로 독립된 리서치·검증·구현을 동시 수행오케스트레이터가 통합
L4 제한 자율반복 운영 작업을 종료조건 안에서 수행고위험 액션은 승인

0. Intake
맥락 수집
문제, 사용자, 비목표, 민감정보 범위누락 질문·리스크·필요 자료 목록화요구·제약·성공 기준이 한 문서에 있음
1. Plan
분해
우선순위와 허용 범위작업을 독립 단위로 쪼개고 위임 레벨 제안작업별 입력/출력/검증자가 보임
2. Execute
생성
고위험 승인 지점초안 작성, 코드 수정, 리서치, 대안 생성산출물이 지정 형식으로 제출됨
3. Verify
검증
최종 판단과 예외 승인테스트, 출처 대조, 스크린샷, 역산, 적대적 리뷰실패 항목이 수정되거나 명시적으로 보류됨
4. Release
배포
배포 승인, 고객 발송, 비용 발생 액션빌드, 스모크 테스트, 릴리스 노트 초안롤백 경로와 확인 URL이 있음
5. Learn
재사용
다음 기준 업데이트 여부실패 원인, 좋은 프롬프트, 체크리스트를 운영 자산화다음 작업에서 재사용할 규칙이 남음

병렬화 규칙

병렬 작업은 쓰기 범위가 겹치지 않을 때만 실행한다. 리서치 에이전트는 결론과 근거 링크만, 구현 에이전트는 수정 파일과 검증 결과만 반환하게 해 메인 컨텍스트를 작게 유지한다.3

검증 규칙

코드는 테스트/린트/스모크, UI는 모바일·데스크톱 스크린샷, 수치는 재계산, 주장은 1차 출처 대조를 기본 게이트로 둔다. AI 결과물은 "정답"이 아니라 검증 대상 초안이다.5,25

운영 로그

프롬프트, 입력 자료, 도구 호출, 변경 diff, 실패 사유, 최종 승인자를 남긴다. 로그가 있어야 나중에 평가셋·가드레일·체크리스트로 환원할 수 있다.25,26

증폭기 가설 (DORA 2025)

Google의 DORA 보고서는 AI를 조직 역량의 증폭기로 규정한다 — 건강한 시스템·프로세스를 가진 팀에선 효과가 크게, 그렇지 못한 팀에선 미미하거나 마이너스로 증폭된다. 즉 ROI는 모델이 아니라 조직 시스템에서 나온다.6

반증·폐기된 주장 (검증의 정직성)

"효과적 에이전틱 코딩엔 작업계약·권한등급·평가·감사로그·롤백의 5대 통제 레이어가 반드시 필요하다"는 한 블로그 주장은 3:0 만장일치로 반증·폐기했다(1차 출처 근거 부재). 그럴듯한 프레임워크라도 출처가 약하면 본 보고서에서 제외했다.

⚖️ 종합 판단 — 모순을 어떻게 읽을까

"90.2% 우수"(vendor 낙관)와 "19% 둔화"(독립 비관)는 충돌이 아니라 조건의 차이다. 전자는 폭넓은 탐색형·병렬화 가능 과제 + 잘 설계된 하네스, 후자는 숙련자가 이미 빠삭한 코드베이스 + 도구 마찰. 결론: 방법론(분해·검증·감독)을 갖추고 적합한 과제에 적용할 때 AI는 강력하며, 무차별 적용은 역효과를 낼 수 있다.

  • METR의 19% 둔화가 2026년 개선된 하네스에서도 재현되는지 미확인.
